# 数据库备份增强功能设计文档

**日期**: 2026-06-15  
**版本**: v1.0  
**项目**: LikeGirl v5.2.3-Stable

---

## 1. 需求概述

在现有数据库备份与恢复功能基础上，新增以下功能：
1. **备份文件删除**：支持单个删除和批量删除备份文件
2. **本地文件上传**：允许管理员上传本地 SQL 备份文件用于恢复
3. **安全加固**：严格的文件验证和路径安全防护

---

## 2. 功能设计

### 2.1 备份文件删除

**单个删除：**
- 在备份列表的每一行"操作"列中添加"删除"按钮
- 点击后弹出确认对话框，确认后删除文件
- 删除成功后刷新列表

**批量删除：**
- 在表格每行首列添加复选框
- 表头添加全选复选框
- 顶部添加"批量删除"按钮（选中项数量 > 0 时启用）
- 批量删除时显示选中数量，确认后统一删除
- 删除完成后刷新列表并清空选中状态

**Why:** 用户积累大量备份文件时，批量删除比逐个删除效率更高；单个删除满足精确操作需求。

**How to apply:** 复用项目中相册管理的批量操作交互模式（复选框 + 批量按钮），保持界面一致性。

### 2.2 本地文件上传

**上传入口：**
- 在"创建新备份"按钮旁添加"上传备份"按钮
- 点击后弹出模态框，包含文件选择器和上传表单
- 支持选择 `.sql` 文件，显示文件名和大小预览
- 上传按钮在选择文件后启用

**上传流程：**
1. 用户选择本地 `.sql` 文件
2. 前端显示文件信息（文件名、大小）
3. 点击上传，AJAX 提交到后端
4. 后端验证文件合法性（见安全设计章节）
5. 验证通过后保存到 `backups/` 目录，自动重命名为 `backup_YYYY-MM-DD_HH-ii-ss.sql` 格式
6. 返回成功信息，前端刷新列表

**Why:** 用户可能在其他环境生成备份文件，或从旧服务器迁移数据，需要上传功能来导入外部备份。

**How to apply:** 使用标准 HTML5 `<input type="file" accept=".sql">` 和 FormData API，无需引入第三方上传库。

### 2.3 UI交互设计

**备份列表表格结构：**
```

----|---------|---------|---------|--------------------------------
[√] | backup_2026-06-15_14-30-00.sql | 125.6 KB | 2026-06-15 14:30:00 | [恢复] [下载] [删除]
[ ] | backup_2026-06-14_10-20-15.sql | 123.4 KB | 2026-06-14 10:20:15 | [恢复] [下载] [删除]
```

**按钮布局：**
```

```

**模态框设计（上传备份）：**
```

         （仅支持 .sql 格式，最大 50MB）
```

## 3. 安全设计

### 3.1 文件上传安全防护

**文件名验证：**
- **白名单验证**：上传后的文件名必须符合 `backup_YYYY-MM-DD_HH-ii-ss.sql` 格式
- **原始文件名过滤**：用户上传的原始文件名仅用于日志记录，不直接用于文件系统操作
- **自动重命名**：服务端接收文件后，使用当前时间戳生成标准格式文件名，覆盖原始文件名

**路径安全：**
- **路径规范化**：使用 `realpath()` 函数验证最终路径必须在 `backups/` 目录内
- **禁止路径遍历**：拒绝包含 `../`、`..\\`、`./` 等路径操作符的文件名
- **目录隔离**：备份文件统一存储在 `backups/` 目录，与代码目录隔离

**文件内容验证：**
- **扩展名检查**：只允许 `.sql` 扩展名
- **MIME类型检查**：验证 `$_FILES['file']['type']` 为 `text/plain`、`application/sql` 或 `application/x-sql`
- **文件头检查**：读取文件前几行，验证是否包含 SQL 关键字（`CREATE`、`INSERT`、`DROP`、`SET` 等）
- **文件大小限制**：限制上传文件最大 50MB，防止资源耗尽攻击

**SQL注入防护：**
- 恢复数据库时，不对 SQL 文件内容做动态拼接
- 使用 `mysqli_multi_query()` 直接执行完整 SQL 文件内容
- 记录恢复操作日志到 `warning` 表

### 3.2 文件删除安全防护

**文件名验证：**
- **白名单验证**：删除的文件名必须匹配 `/^backup_\d{4}-\d{2}-\d{2}_\d{2}-\d{2}-\d{2}\.sql$/` 正则
- **路径规范化**：使用 `realpath()` 确保目标文件在 `backups/` 目录内

**权限验证：**
- 所有操作必须验证 `$_SESSION['loginadmin']` 存在
- 操作失败时返回 JSON 错误，不暴露服务器路径

**批量删除防护：**
- 单次批量删除最多 100 个文件，防止误操作
- 删除前二次确认，前端显示文件数量

### 3.3 会话安全

**会话验证：**
- 所有后端接口开头执行 `session_start()` 和 `$_SESSION['loginadmin']` 检查
- 未授权访问返回 `{'success': false, 'message': '未授权'}`

**CSRF防护（可选增强）：**
- 当前系统未实现 CSRF Token，建议后续版本添加
- 临时方案：限制操作频率（同一会话 10 秒内最多 1 次删除/上传操作）

### 3.4 错误处理

**统一错误返回格式：**
```json
{
  "success": false,
  "message": "用户友好的错误信息"
}
```

**错误信息安全原则：**
- 不暴露服务器路径、数据库信息
- 不返回详细的 PHP 错误堆栈
- 记录详细错误到服务器日志，前端只显示通用提示

## 5. 接口设计

### 5.1 删除备份文件接口

**文件**: `admin/deleteBackup.php`

**请求方式**: POST

**请求参数**:
```json
{
  "filenames": ["backup_2026-06-15_14-30-00.sql", "backup_2026-06-14_10-20-15.sql"]
}
```

**返回格式**:
```json
{
  "success": true,
  "message": "成功删除 2 个备份文件",
  "deleted": 2
}
```

**错误示例**:
```json
{
  "success": false,
  "message": "部分文件删除失败",
  "deleted": 1,
  "failed": 1
}
```

**安全检查流程**:
1. 验证会话 `$_SESSION['loginadmin']`
2. 验证 `filenames` 参数为数组且不为空
3. 限制单次最多删除 100 个文件
4. 遍历每个文件名：
   - 正则验证格式
   - 路径规范化检查
   - 文件存在性检查
   - 执行 `unlink()` 删除
5. 记录操作日志到 `warning` 表
6. 返回删除结果

### 5.2 上传备份文件接口

**文件**: `admin/uploadBackup.php`

**请求方式**: POST (multipart/form-data)

**请求参数**:
```
file: <SQL文件>
```

**返回格式**:
```json
{
  "success": true,
  "message": "上传成功",
  "filename": "backup_2026-06-15_15-30-00.sql",
  "size": 2408960
}
```

**错误示例**:
```json
{
  "success": false,
  "message": "文件格式不正确，仅支持 .sql 文件"
}
```

**安全检查流程**:
1. 验证会话 `$_SESSION['loginadmin']`
2. 验证 `$_FILES['file']` 存在且无上传错误
3. 检查文件大小（最大 50MB）
4. 验证文件扩展名为 `.sql`
5. 验证 MIME 类型
6. 读取文件前 1KB，检查是否包含 SQL 关键字
7. 生成标准格式文件名 `backup_YYYY-MM-DD_HH-ii-ss.sql`
8. 使用 `move_uploaded_file()` 移动到 `backups/` 目录
9. 验证最终路径在 `backups/` 目录内（`realpath()` 检查）
10. 记录操作日志到 `warning` 表
11. 返回上传结果

### 5.3 备份列表接口（修改）

**文件**: `admin/backupList.php`

**修改点**: 无需修改，现有接口已满足需求

## 8. 部署注意事项

### 8.1 PHP配置检查

确保 `php.ini` 配置支持文件上传：
```ini
file_uploads = On
upload_max_filesize = 50M
post_max_size = 50M
max_execution_time = 300
memory_limit = 256M
```

### 8.2 目录权限

确保 `backups/` 目录可读写：
```bash
chmod 755 backups/
```

### 8.3 备份目录清理策略（可选）

建议添加定时任务，自动清理 30 天前的备份文件：
```bash
# crontab -e
0 2 * * * find /path/to/backups -name "backup_*.sql" -mtime +30 -delete
```

## 10. 风险评估

| 风险项 | 严重程度 | 缓解措施 |
|-------|---------|---------|
| 恶意文件上传导致服务器被控制 | 高 | 严格的文件验证 + 自动重命名 + 路径隔离 |
| 路径遍历攻击删除系统文件 | 高 | realpath() 验证 + 白名单文件名 |
| SQL注入攻击 | 中 | 恢复操作记录日志 + 定期备份 |
| 超大文件上传导致磁盘满 | 中 | 文件大小限制 + 磁盘空间监控 |
| 误删除重要备份 | 中 | 二次确认对话框 + 操作日志记录 |
